# ДП на матрицах. Кол-во путей в графе из u в v длины <= k

## 1. Как эта задача вырастает из предыдущей

В предыдущем вопросе мы научились считать:

> сколько существует путей длины ровно `k`.

Теперь задача чуть тоньше:

> сколько существует путей длины не больше `k`.

На первый взгляд различие небольшое, но алгоритмически это уже не одна степень
матрицы, а **сумма степеней**.

## 2. От “ровно” к “не больше”

Если:

```text
A^t[u][v]
```

равно числу путей длины ровно `t`, то число путей длины не больше `k` — это:

```text
I + A + A^2 + ... + A^k
```

Где:

- `I` — единичная матрица;
- она отвечает за путь длины 0.

## 3. Почему единичная матрица отвечает длине 0

Путь длины 0 — это “остаться в той же вершине и никуда не идти”.

Значит:

- из вершины `u` в неё же есть ровно один путь длины 0;
- в любую другую вершину — ни одного.

Это ровно и есть единичная матрица:

```text
I[u][u] = 1
I[u][v] = 0, если u != v
```

## 4. Почему сумма степеней корректна

Каждая степень отвечает за свою конкретную длину:

- `A^0` — длина 0;
- `A^1` — длина 1;
- `A^2` — длина 2;
- ...
- `A^k` — длина `k`.

Если нас интересуют все длины до `k`, то нужно сложить все эти вклады.

Это почти буквально то же самое, что в обычной динамике:

```text
answer = sum(dp[len][v]) по всем len от 0 до k
```

Просто здесь всё записано матрично.

## 5. Как это связано с динамикой по длине

Можно было бы строить таблицу:

```text
dp[len][u][v]
```

или, более компактно:

```text
dp[len] = матрица числа путей длины len
```

Тогда:

```text
sum_{t=0}^{k} dp[t]
```

и есть искомая матрица ответов.

Матричная форма не меняет идею, а делает её более компактной и пригодной для
ускорения.

## 6. Почему задача уже чуть сложнее, чем “ровно `k`”

Для “ровно `k`” нам достаточно одной степени:

```text
A^k
```

Для “не больше `k`” нужна сумма:

```text
I + A + A^2 + ... + A^k
```

Значит, если делать всё совсем в лоб:

- либо считать все степени по очереди;
- либо придумать более умную технику суммирования вместе с возведением.

## 7. Наивный способ

Можно сделать так:

1. взять единичную матрицу `I`;
2. последовательно перемножать степени;
3. суммировать их.

Это уже работает, но не так красиво и быстро, как хотелось бы.

## 8. Идея блочной матрицы

Есть продвинутая техника, позволяющая одновременно получить и `A^k`, и сумму
степеней через одну степень блочной матрицы.

Например, можно использовать матрицу вида:

```text
[A I]
[0 I]
```

При возведении в степень в верхнем правом блоке аккумулируется сумма степеней.

Это уже более теоретический приём, но очень полезно знать, что такая связка
существует.

## 9. Почему это хороший пример агрегирования по длинам

Тема важна не только сама по себе. Она формирует полезную привычку:

если задача спрашивает не одну длину, а диапазон длин, нужно понять:

- как объединяются ответы по отдельным длинам;
- можно ли агрегировать их структурно, а не просто перебирать.

## 10. Где встречаются такие постановки

Формулировки типа:

- “за не более чем `k` шагов”;
- “за максимум `k` переходов”;
- “до глубины `k`”;
- “в пределах `k` преобразований”

очень часто приводят именно к сумме степеней переходной матрицы.

## 11. В чём математическая красота идеи

Для длины ровно `k` нам нужна одна степень.
Для длины не больше `k` — геометрическая сумма матриц.

Это почти прямой аналог школьной алгебры:

```text
1 + x + x^2 + ... + x^k
```

Только вместо числа `x` теперь стоит матрица переходов.

## 12. Реализация простого варианта на C++

```cpp
using Matrix = std::vector<std::vector<long long>>;

Matrix Multiply(const Matrix& a, const Matrix& b) {
  int n = static_cast<int>(a.size());
  Matrix c(n, std::vector<long long>(n, 0));
  for (int i = 0; i < n; ++i) {
    for (int k = 0; k < n; ++k) {
      if (a[i][k] == 0) {
        continue;
      }
      for (int j = 0; j < n; ++j) {
        c[i][j] += a[i][k] * b[k][j];
      }
    }
  }
  return c;
}

Matrix Add(const Matrix& a, const Matrix& b) {
  int n = static_cast<int>(a.size());
  Matrix c = a;
  for (int i = 0; i < n; ++i) {
    for (int j = 0; j < n; ++j) {
      c[i][j] += b[i][j];
    }
  }
  return c;
}

Matrix Identity(int n) {
  Matrix id(n, std::vector<long long>(n, 0));
  for (int i = 0; i < n; ++i) {
    id[i][i] = 1;
  }
  return id;
}

Matrix CountPathsAtMostKNaive(const Matrix& adjacency, int k) {
  int n = static_cast<int>(adjacency.size());
  Matrix result = Identity(n);
  Matrix current = Identity(n);

  for (int step = 1; step <= k; ++step) {
    current = Multiply(current, adjacency);
    result = Add(result, current);
  }

  return result;
}
```

Это не самый быстрый подход, но он очень хорошо иллюстрирует идею.

## 13. Сложность простого подхода

Если:

- каждое умножение матриц стоит `O(n^3)`;
- делаем это `k` раз,

то получаем:

```text
O(k n^3)
```

Это приемлемо, если `k` небольшое, но для больших `k` хочется использовать
более умные техники.

## 14. Сравнение двух постановок

| Вопрос | Матрический объект |
|---|---|
| Сколько путей длины ровно `k`? | `A^k` |
| Сколько путей длины не больше `k`? | `I + A + A^2 + ... + A^k` |

Это очень полезная таблица для распознавания задач.

## 15. Концептуальная связь с достижимостью

Если вместо обычной арифметики перейти к булевой, та же схема начинает решать:

- достижимость за ровно `k` шагов;
- достижимость за не более чем `k` шагов.

То есть меняется не каркас задачи, а алгебра, в которой мы работаем.

## 16. Типичные ошибки

- забыть `A^0`;
- считать, что `A^k` уже включает пути меньших длин;
- путать “не больше `k`” с “ровно `k`”;
- не понимать, почему в сумму входит единичная матрица;
- случайно удвоить или пропустить какие-то длины в суммировании.

## 17. Что важно запомнить

Задача про пути длины `<= k` — это естественное продолжение задачи про пути
длины ровно `k`.

Главная идея:

1. одна степень матрицы отвечает одной длине;
2. диапазон длин требует суммы степеней;
3. матричное `DP` умеет не только моделировать переход, но и агрегировать
   результаты по длинам.

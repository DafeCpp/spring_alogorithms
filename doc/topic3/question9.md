# Основные операции над декартовым деревом

## 1. Почему операции в treap считаются особенно красивыми

Многие деревья поиска поддерживают операции через длинные наборы частных
случаев.

У `treap` есть очень сильное преимущество:

почти вся структура операций выражается через два фундаментальных примитива:

- `split`;
- `merge`.

Именно это делает treap настолько любимым в задачах и расширениях.

---

## 2. Операция split: формальная цель

`split(T, x)` разбивает дерево `T` на два дерева:

- `L` — все ключи `< x`;
- `R` — все ключи `>= x`.

Иногда используют вариант:

- `<= x` и `> x`.

Это просто вопрос договорённости. Главное — быть последовательным.

---

## 3. Интуиция split

Split — это “разрез дерева по ключевому порогу”.

Мы как будто говорим:

> Всё, что меньше `x`, живёт слева. Всё, что не меньше, живёт справа.

Красота в том, что дерево не нужно расплющивать в массив и собирать заново.
Разделение делается рекурсивно по самой структуре.

---

## 4. Как split работает рекурсивно

Пусть корень дерева имеет ключ `key`.

### Случай 1. `key < x`

Тогда:

- корень точно должен остаться в левой части;
- возможно, часть его правого поддерева тоже пойдёт в левую часть, а часть —
  в правую.

Значит нужно рекурсивно split-ить правое поддерево.

### Случай 2. `key >= x`

Тогда:

- корень должен уйти в правую часть;
- нужно split-ить левое поддерево.

Это типичная очень чистая рекурсивная конструкция.

---

## 5. Почему split сохраняет BST-инвариант

Потому что мы никогда не нарушаем порядок по ключам:

- если вершина осталась слева, то она действительно меньше порога;
- если справа — действительно не меньше;
- пересборка детей идёт только в местах, где это согласовано с порогом.

Иными словами, split не “мешает элементы”, а просто аккуратно разносит по двум
корректным BST-деревьям.

---

## 6. Операция merge: формальная цель

`merge(L, R)` склеивает два дерева при условии:

- все ключи в `L` меньше всех ключей в `R`.

Если это условие нарушено, обычный `merge` делать нельзя, иначе сломается
инвариант дерева поиска.

---

## 7. Интуиция merge

Нужно выбрать, кто станет корнем объединённого дерева.

Раз по приоритетам treap — это куча, то корнем должен стать тот из двух корней,
у кого приоритет лучше.

Получаются два случая:

- либо корень `L` выше по приоритету;
- либо корень `R`.

После этого недостающий ребёнок строится рекурсивным merge.

Это невероятно красивая идея:

локальное решение “кто корень” автоматически задаёт рекурсивную форму всего
остального дерева.

---

## 8. Почему merge сохраняет оба инварианта

### Инвариант по ключам

Он сохраняется, потому что merge разрешён только когда:

- все ключи в `L` меньше всех ключей в `R`.

### Инвариант по приоритетам

Он сохраняется, потому что в корень всегда поднимается вершина с лучшим
приоритетом среди двух корней.

То есть:

- поиск защищается условием на ключи;
- форма защищается условием на приоритеты.

---

## 9. Поиск в treap

Поиск никак не использует приоритеты напрямую.

Он идёт как в обычном `BST`:

1. сравнить ключ;
2. пойти влево или вправо.

Средняя сложность:

```text
O(log n)
```

потому что средняя высота treap логарифмическая.

---

## 10. Вставка: два взгляда

Есть два одинаково полезных способа мыслить о вставке.

### Взгляд 1. BST + подъём по приоритету

1. вставляем вершину как в обычный `BST`;
2. если её приоритет лучше приоритета родителя, поднимаем вершину вращениями.

Это хороший мост к пониманию AVL и red-black.

### Взгляд 2. split + merge

1. делим дерево по ключу новой вершины;
2. создаём новую вершину;
3. склеиваем `L`, новую вершину и `R`.

Этот взгляд особенно красив и важен в олимпиадной практике.

---

## 11. Почему вставка через split/merge так естественна

Потому что новая вершина с данным ключом должна стоять ровно между:

- всеми меньшими ключами;
- всеми большими или равными.

Split как раз и строит эти две части автоматически.

То есть операция:

```text
insert(x) = merge(merge(L, new_node), R)
```

не просто красива синтаксически, а идеально соответствует логике порядка.

---

## 12. Удаление

Удаление в treap тоже выглядит очень чисто.

Если мы нашли вершину:

- её левое поддерево содержит все меньшие ключи;
- правое — все большие.

Значит вместо удаляемой вершины можно просто поставить:

```text
merge(left_subtree, right_subtree)
```

Вот это и есть одна из самых элегантных причин любить treap.

---

## 13. Почему удаление через merge корректно

Потому что:

- все ключи слева уже меньше всех ключей справа;
- merge сохранит и порядок по ключам, и кучу по приоритетам.

То есть удаление сводится не к длинной ручной перестройке, а к одной аккуратной
базовой операции.

---

## 14. Сложности операций

При случайных приоритетах в среднем:

- поиск: `O(log n)`;
- вставка: `O(log n)`;
- удаление: `O(log n)`;
- `split`: `O(log n)`;
- `merge`: `O(log n)`.

Почему:

- все операции движутся вдоль одного пути или его рекурсивного аналога;
- средняя высота treap логарифмическая.

## 15. Реализация на C++

```cpp
struct Node {
  int key;
  int priority;
  Node* left = nullptr;
  Node* right = nullptr;
};

void Split(Node* root, int key, Node*& left, Node*& right) {
  if (root == nullptr) {
    left = nullptr;
    right = nullptr;
    return;
  }

  if (root->key < key) {
    Split(root->right, key, root->right, right);
    left = root;
  } else {
    Split(root->left, key, left, root->left);
    right = root;
  }
}

Node* Merge(Node* left, Node* right) {
  if (left == nullptr) {
    return right;
  }
  if (right == nullptr) {
    return left;
  }

  if (left->priority > right->priority) {
    left->right = Merge(left->right, right);
    return left;
  }

  right->left = Merge(left, right->left);
  return right;
}

Node* Find(Node* root, int key) {
  if (root == nullptr || root->key == key) {
    return root;
  }
  if (key < root->key) {
    return Find(root->left, key);
  }
  return Find(root->right, key);
}

Node* Insert(Node* root, Node* node) {
  if (root == nullptr) {
    return node;
  }

  if (node->priority > root->priority) {
    Split(root, node->key, node->left, node->right);
    return node;
  }

  if (node->key < root->key) {
    root->left = Insert(root->left, node);
  } else {
    root->right = Insert(root->right, node);
  }
  return root;
}

Node* Erase(Node* root, int key) {
  if (root == nullptr) {
    return nullptr;
  }

  if (root->key == key) {
    Node* merged = Merge(root->left, root->right);
    delete root;
    return merged;
  }

  if (key < root->key) {
    root->left = Erase(root->left, key);
  } else {
    root->right = Erase(root->right, key);
  }
  return root;
}
```

---

## 16. Почему treap так удобен для range-операций

Как только у нас есть `split` и `merge`, очень легко:

- выделить диапазон ключей;
- отрезать префикс;
- отрезать суффикс;
- временно изолировать подмножество, обработать его и склеить обратно.

Именно отсюда treap получает огромную популярность в задачах на последовательности.

---

## 17. Неявное декартово дерево

Есть важное расширение: ключом может быть не “значение множества”, а
**позиция элемента в последовательности**.

Тогда treap превращается в структуру, которая умеет:

- разрезать массив по позиции;
- склеивать куски;
- двигать отрезки;
- делать операции над диапазонами.

Это уже не просто дерево поиска, а полноценный мощный контейнер для
последовательностей.

---

## 18. Сравнение с AVL и red-black по операциям

| Структура | Базовый стиль операций |
|---|---|
| AVL | поиск как в BST + жёсткие локальные вращения |
| Red-black | поиск как в BST + вращения и перекрашивания |
| Treap | поиск как в BST + split/merge или подъём по приоритету |

То есть treap особенно красив именно **алгеброй операций**.

---

## 19. Что важно запомнить

Если ты хорошо понимаешь:

1. `split`,
2. `merge`,
3. вставку как `split + new node + merge`,
4. удаление как `merge(left, right)`,

то ты уже понимаешь почти всю механику treap.

Именно поэтому декартово дерево часто кажется самым “прозрачным” из всех
сбалансированных деревьев курса.

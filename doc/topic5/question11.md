# ДП на матрицах. Есть ли хотя бы один путь из u в v длины ровно k?

## 1. Почему эта задача принципиально отличается от подсчёта количества

До этого мы спрашивали:

> сколько существует путей?

Теперь вопрос другой:

> существует ли хотя бы один путь?

Это кажется маленькой правкой текста, но на самом деле меняет саму алгебру
задачи.

Если раньше нас интересовало точное число способов, теперь нас интересует лишь
факт:

- `да`;
- `нет`.

## 2. Почему обычная арифметика здесь избыточна

Если есть 1 путь и если есть 1000 путей, для текущей постановки это одно и то
же:

```text
истина
```

Значит нам не нужно:

- накапливать большие числа;
- различать количества;
- считать точную кратность.

Нужно только уметь отвечать:

> достижимо или нет?

## 3. Булева алгебра как естественная модель

В обычном матричном умножении используются операции:

- сложение `+`;
- умножение `*`.

Для булевой версии мы заменяем их на:

- `OR` вместо `+`;
- `AND` вместо `*`.

Почему это естественно:

- чтобы существовал путь через некоторую промежуточную вершину, нужно, чтобы
  существовал путь на первом участке **и** на втором;
- чтобы существовал путь вообще, достаточно хотя бы одного подходящего
  промежуточного варианта.

То есть:

```text
AND = оба куска существуют
OR  = существует хотя бы один способ собрать путь
```

## 4. Булево матричное умножение

Если `B` и `C` — булевы матрицы, то:

```text
R[i][j] = OR по всем t (B[i][t] AND C[t][j])
```

Интерпретация очень прозрачная:

> существует ли такая промежуточная вершина `t`, что путь из `i` в `t`
> существует, и путь из `t` в `j` тоже существует?

Это точный аналог обычного матричного умножения, только в мире логики.

## 5. Что означает степень булевой матрицы

Если `A` — булева матрица смежности графа, то:

```text
A^k[u][v] = true
```

тогда и только тогда, когда существует хотя бы один путь длины ровно `k`
из `u` в `v`.

Это важнейший факт темы.

## 6. Почему это верно

### База

Для `k = 1` всё очевидно:

```text
A[u][v] = true
```

означает наличие ребра `u -> v`.

### Переход

Пусть степень `k` уже интерпретируется правильно.
Тогда степень `k + 1` проверяет:

есть ли вершина `t`, такая что:

- существует путь длины `k` из `u` в `t`;
- существует ребро `t -> v`.

Именно так и устроен путь длины `k + 1`.

То есть логика полностью совпадает с обычной динамикой по числу шагов.

## 7. Сравнение с задачей на количество путей

| Вопрос | Алгебра | Что хранится |
|---|---|---|
| Сколько путей длины `k`? | обычная | число способов |
| Есть ли путь длины `k`? | булева | факт достижимости |

Это очень полезное сравнение:

> структура переходов остаётся той же, а меняется смысл операций и хранимой
> информации.

## 8. Связь с обычной динамикой по длине

Можно ввести:

```text
dp[len][v] = существует ли путь длины len, оканчивающийся в v
```

Тогда переход выглядит так:

```text
dp[len + 1][to] = OR по всем from:
                  dp[len][from] AND edge[from][to]
```

Булево матричное умножение — это просто компактная запись этого перехода для
всех вершин сразу.

## 9. Булева матрица смежности

В булевой версии:

```text
A[i][j] = true, если есть ребро i -> j
A[i][j] = false, если ребра нет
```

Дальше все вычисления ведутся не над числами, а над логическими значениями.

## 10. Реализация на C++

```cpp
using BoolMatrix = std::vector<std::vector<bool>>;

BoolMatrix MultiplyBool(const BoolMatrix& a, const BoolMatrix& b) {
  int n = static_cast<int>(a.size());
  BoolMatrix c(n, std::vector<bool>(n, false));

  for (int i = 0; i < n; ++i) {
    for (int k = 0; k < n; ++k) {
      if (!a[i][k]) {
        continue;
      }
      for (int j = 0; j < n; ++j) {
        c[i][j] = c[i][j] || (a[i][k] && b[k][j]);
      }
    }
  }

  return c;
}

BoolMatrix PowerBool(BoolMatrix base, long long exp) {
  int n = static_cast<int>(base.size());
  BoolMatrix result(n, std::vector<bool>(n, false));
  for (int i = 0; i < n; ++i) {
    result[i][i] = true;
  }

  while (exp > 0) {
    if (exp & 1LL) {
      result = MultiplyBool(result, base);
    }
    base = MultiplyBool(base, base);
    exp >>= 1LL;
  }

  return result;
}

bool HasPathExactlyK(const BoolMatrix& adjacency, int u, int v, long long k) {
  BoolMatrix p = PowerBool(adjacency, k);
  return p[u][v];
}
```

## 11. Почему единичная матрица остаётся важной

Даже в булевой версии:

- `A^0 = I`;
- это означает путь длины 0.

То есть:

- из вершины в саму себя путь длины 0 существует;
- из вершины в другую — нет.

Это тот же самый смысл, что и в обычной арифметике.

## 12. Почему эта тема очень важна концептуально

Она показывает крайне глубокую идею:

> один и тот же алгоритмический каркас можно использовать в разных “алгебрах”.

В зависимости от задачи мы можем заменить:

- `+` на `OR`;
- `*` на `AND`;
- числа на логические значения.

И при этом сохранить смысл степеней матрицы как многократного применения
перехода.

## 13. Связь с автоматами и достижимостью

Булевы матричные переходы полезны не только в графах.
Они встречаются и в:

- автоматах;
- системах переходов;
- моделях достижимости;
- задачах на достижение состояния за фиксированное число шагов.

То есть это уже не только частная техника, а довольно универсальный способ
мыслить о дискретных переходах.

## 14. Что делать, если нужно “не больше k”

Тогда, как и в численной версии, нужно не только `A^k`, а агрегирование
степеней:

- либо логическое `OR` по степеням от `0` до `k`;
- либо более хитрая матричная конструкция.

То есть переход от “ровно” к “не больше” здесь тоже сохраняется, просто теперь
вместо сумм используется булево объединение.

## 15. Типичные ошибки

- использовать обычное матричное умножение вместо булевого;
- думать, что если путей два, то это “лучше”, хотя для булевой задачи это не
  имеет значения;
- забыть про путь длины 0 и единичную матрицу;
- путать задачу на достижимость с задачей на подсчёт.

## 16. Что важно запомнить

Булева матрица переходов — это очень мощное обобщение обычной динамики:

1. состояние по-прежнему кодирует достижимость;
2. переход по-прежнему означает один шаг;
3. степень матрицы по-прежнему означает много шагов;
4. меняется только алгебра: вместо подсчёта количества мы проверяем
   существование.

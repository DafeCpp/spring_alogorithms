# Нижняя оценка времени работы для сортировок сравнением

## Что именно мы хотим доказать

Если алгоритм сортирует элементы **только с помощью сравнений**, то он не может
работать быстрее, чем `Ω(n log n)` в худшем случае.

Это означает, что для сравнительных сортировок `merge sort`, `heap sort` и
`quick sort` среднего уровня сложности уже почти оптимальны.

## Дерево решений

Любую сравнительную сортировку можно представить в виде бинарного дерева:

- внутренняя вершина — сравнение `a[i] < a[j]?`;
- ребро — результат сравнения;
- лист — итоговый порядок элементов.

## Почему листьев должно быть много

Если все элементы различны, существует `n!` возможных перестановок входа.
Чтобы алгоритм отличал их друг от друга, в дереве решений должно быть не меньше
`n!` листьев.

## Связь с высотой дерева

У бинарного дерева высоты `h` число листьев не превосходит `2^h`. Значит:

```text
2^h >= n!
```

Берём логарифм:

```text
h >= log2(n!)
```

А по оценке Стирлинга:

```text
log2(n!) = Ω(n log n)
```

## Вывод

Любая сортировка, которая опирается только на сравнения, обязана в худшем случае
делать хотя бы порядка `n log n` шагов сравнения.

## Почему это важно

Это объясняет, почему:

- нельзя мечтать о сравнительной сортировке за `O(n)` для произвольных чисел;
- но можно получить `O(n)` для специальных случаев, например у counting sort,
  потому что она не является чисто сравнительной.

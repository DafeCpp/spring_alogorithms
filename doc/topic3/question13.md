# Красно-чёрное дерево

## 1. Зачем понадобилось красно-чёрное дерево

AVL-дерево даёт очень строгий баланс, но эта строгость делает поддержку формы
более чувствительной.

Красно-чёрное дерево — это другой ответ на ту же проблему:

- не контролировать разницу высот напрямую;
- а навязать более мягкие структурные инварианты через цвета.

Идея такая:

> пусть дерево не идеально симметрично, но зато гарантированно не слишком
> высоко.

---

## 2. Что такое red-black дерево

Это бинарное дерево поиска, в котором у каждой вершины есть цвет:

- красный;
- чёрный.

Цвет — это не часть пользовательских данных, а служебная информация, которая
кодирует ограничения на форму дерева.

---

## 3. Стандартные инварианты

Обычно формулируют так:

1. каждая вершина либо красная, либо чёрная;
2. корень чёрный;
3. все `null`-листья считаются чёрными;
4. у красной вершины не может быть красного ребёнка;
5. на всех путях от вершины до её `null`-листьев одинаковое число чёрных
   вершин.

Пункты 4 и 5 — самые содержательные.

Именно они реально ограничивают форму дерева.

---

## 4. Почему эти правила вообще полезны

Если смотреть на них по отдельности, они могут казаться странными.

Но вместе они дают очень важный эффект:

- красные вершины не могут образовывать длинные плотные цепочки;
- чёрные вершины распределены по путям равномерно.

Именно это удерживает дерево от сильного перекоса.

---

## 5. Что такое чёрная высота

Ключевое понятие темы:

```text
black height
```

Это число чёрных вершин на любом пути от текущей вершины до листа.

Так как инвариант требует одинакового количества чёрных вершин на всех путях,
чёрная высота вершины определена корректно.

Это понятие играет для red-black дерева примерно ту же роль, что обычная высота
для AVL.

---

## 6. Интуитивная философия red-black

Если AVL говорит:

> локальные высоты поддеревьев должны быть почти равны,

то red-black говорит:

> длины путей могут отличаться, но “структурная чёрная масса” должна быть
> распределена ровно, а красные уровни не должны идти подряд.

То есть red-black — это более мягкая, но всё ещё очень сильная форма
балансировки.

---

## 7. Почему красные вершины не могут идти подряд

Если разрешить длинные красные цепочки, можно было бы получить очень длинные
пути, почти не увеличивая чёрную высоту.

Запрет на двух красных подряд означает:

- красный уровень можно вставить между чёрными;
- но нельзя бесконечно наращивать “тонкую” красную лестницу.

Из этого и рождается ограничение общей высоты.

---

## 8. Почему дерево остаётся близко к логарифмическому

На любом пути:

- чёрные вершины контролируются строго;
- красных между ними может быть не больше одной подряд.

Значит длина пути не может быть больше примерно удвоенной чёрной высоты.

А чёрная высота, в свою очередь, связана логарифмически с числом вершин.

Именно отсюда появляется:

```text
height = O(log n)
```

---

## 9. Чем red-black отличается от AVL по духу

| Вопрос | AVL | Red-black |
|---|---|---|
| Что контролируется напрямую | разница высот | цветовые инварианты |
| Жёсткость формы | выше | мягче |
| Поиск | обычно чуть лучше | тоже `O(log n)` |
| Поддержка | более строгая | часто удобнее в общих контейнерах |

Очень важно видеть: red-black не “хуже AVL”, а просто выбирает другой
компромисс между строгостью и удобством поддержки.

---

## 10. Почему эта структура так популярна в библиотеках

Потому что она даёт сильный инженерный компромисс:

- худшие гарантии `O(log n)` есть;
- дерево не слишком сложно поддерживать;
- число локальных перестроек разумное;
- структура хорошо подходит для ordered container общего назначения.

Именно поэтому реализация `map` и `set` во многих стандартных библиотеках
ассоциируется именно с red-black деревьями.

## 11. Учебный каркас на C++

```cpp
enum class Color {
  Red,
  Black
};

struct Node {
  int key;
  Color color = Color::Red;
  Node* left = nullptr;
  Node* right = nullptr;
  Node* parent = nullptr;
};

void RotateLeft(Node*& root, Node* x) {
  Node* y = x->right;
  x->right = y->left;
  if (y->left != nullptr) {
    y->left->parent = x;
  }

  y->parent = x->parent;
  if (x->parent == nullptr) {
    root = y;
  } else if (x == x->parent->left) {
    x->parent->left = y;
  } else {
    x->parent->right = y;
  }

  y->left = x;
  x->parent = y;
}

void RotateRight(Node*& root, Node* y) {
  Node* x = y->left;
  y->left = x->right;
  if (x->right != nullptr) {
    x->right->parent = y;
  }

  x->parent = y->parent;
  if (y->parent == nullptr) {
    root = x;
  } else if (y == y->parent->left) {
    y->parent->left = x;
  } else {
    y->parent->right = x;
  }

  x->right = y;
  y->parent = x;
}
```

## 12. О чём нельзя думать при изучении red-black

Плохой подход:

- заучивать цветовые случаи как магический ритуал.

Хороший подход:

- помнить, что все операции чинят только две вещи:
  1. запрет двух красных подряд;
  2. равенство чёрных высот.

Тогда даже сложные случаи удаления выглядят уже не случайным набором правил, а
техническими способами восстановления инвариантов.

---

## 13. Ментальная модель, которую важно вынести

Правильный внутренний образ red-black дерева такой:

> Это BST, в котором цвета кодируют ограничение на форму, а не “украшают”
> вершины.

Красный цвет говорит:

- здесь можно немного “сжать” дерево;

чёрный цвет говорит:

- этот уровень обязателен в структуре всех путей.

---

## 14. Что важно запомнить

Red-black дерево нужно понимать через три идеи:

1. это `BST`;
2. цветовые инварианты ограничивают форму;
3. чёрная высота — главный скрытый параметр, из которого рождается
   логарифмическая высота дерева.

Если эти идеи ясны, то вставка, удаление и доказательство высоты становятся
намного менее мистическими.

# Поддержка минимума в стеке

## Задача

Нужно реализовать стек, который кроме стандартных операций умеет быстро отвечать
на запрос:

```text
get_min()
```

и делает это за `O(1)`.

## Почему нельзя просто искать минимум каждый раз

Если после каждого запроса `get_min()` проходить по всему стеку, получится
`O(n)`. Это слишком дорого для большого числа запросов.

## Классическая идея

Храним два стека:

1. основной стек значений;
2. стек минимумов.

### Правило для `push(x)`

- кладём `x` в основной стек;
- если стек минимумов пуст или `x <= current_min`, кладём `x` и туда.

### Правило для `pop()`

- снимаем элемент с основного стека;
- если он совпадает с текущим минимумом, снимаем и вершину стека минимумов.

## Пример

Основной стек:

```text
[3, 1, 4, 1, 5]
```

Стек минимумов:

```text
[3, 1, 1]
```

После удаления верхнего `5` минимум не меняется.
После удаления верхнего `1` снимается и верхний минимум, и новым минимумом
становится предыдущая `1`.

## Альтернативная реализация

Можно хранить не только значения, а пары:

```text
(value, min_on_prefix)
```

Тогда каждая вершина сразу знает минимум на всём стеке под собой.

## Сложность

Все операции:

- `push`;
- `pop`;
- `top`;
- `get_min`

работают за `O(1)`.

## Сводная таблица практических характеристик

Эта структура ускоряет именно запрос минимума. Максимум она не делает быстрым
автоматически.

| Свойство | Оценка |
|---|---|
| Память | `O(n)` для основного стека + до `O(n)` для стека минимумов |
| Узнать длину | `O(1)`, если хранится `size`; иначе `O(n)` |
| Взять максимум | `O(n)` |
| Добавить элемент `push` | `O(1)` |
| Взять минимум `get_min` | `O(1)` |

## Где это используется

- очередь с минимумом через два стека;
- задачи со скользящим окном;
- структуры для онлайн-обработки запросов.

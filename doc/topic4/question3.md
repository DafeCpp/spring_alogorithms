# Полиномиальная хеш-функция. Её использование для строк. Метод Горнера

## 1. Почему строкам нужен специальный хеш

Для целых чисел можно обсуждать:

- остаток от деления;
- мультипликативную схему;
- смешивание битов.

Но строка — это уже не одно число, а последовательность символов.
И для строки важно не только множество символов, но и их порядок.

Например:

```text
abc
acb
bac
```

содержат похожие символы, но это разные строки и их хеши тоже должны сильно
отличаться.

Именно поэтому для строк очень естественно использовать полиномиальный хеш.

## 2. Идея полиномиального хеша

Каждый символ строки можно рассматривать как число:

- код символа;
- номер буквы в алфавите;
- любое фиксированное отображение.

Тогда строку:

```text
s = s[0] s[1] ... s[n-1]
```

можно превратить в полином:

```text
h(s) = s[0] * p^0 + s[1] * p^1 + ... + s[n-1] * p^(n-1)
```

Обычно всё считается по модулю большого числа.

## 3. Почему это похоже на запись числа в системе счисления

Это очень полезная интуиция.

Обычное десятичное число:

```text
472 = 4 * 10^2 + 7 * 10^1 + 2 * 10^0
```

Полиномиальный хеш делает почти то же самое:

- символы играют роль “цифр”;
- `p` играет роль основания.

Только дальше мы ещё берём всё по модулю.

## 4. Почему порядок символов учитывается автоматически

Если поменять символы местами, они окажутся при других степенях `p`.
Поэтому:

```text
abc
```

и

```text
acb
```

получат разные значения почти всегда.

Это и делает полиномиальный хеш особенно естественным для строк:

он не просто суммирует символы, а учитывает их позиции.

## 5. Зачем нужен модуль

Без модуля значения росли бы очень быстро:

- степени `p` становятся огромными;
- числа плохо помещаются в стандартные типы;
- вычисления становятся непрактичными.

Модуль нужен, чтобы:

- ограничивать размер чисел;
- позволить работать с длинными строками;
- сохранить удобные алгебраические свойства.

Обычно выбирают большой модуль, например большое простое число.

## 6. Зачем нужно основание `p`

Основание определяет “вес” позиции символа.

Если выбрать `p` слишком неудачно, разные строки могут чаще сталкиваться.
Если выбрать разумно, хеши распределяются лучше.

На практике `p` обычно:

- больше размера алфавита;
- или хотя бы сравним с ним;
- и выбрано так, чтобы вычисления были удобными.

## 7. Почему это называется полиномиальным хешем

Потому что строка интерпретируется как значение многочлена в точке `p` по модулю
`m`.

Это не просто красивая терминология: именно из этого вырастают:

- префиксные хеши;
- быстрые хеши подстрок;
- алгоритм Рабина — Карпа;
- двойное хеширование строк.

## 8. Метод Горнера

Если считать полином “в лоб”, придётся:

- отдельно возводить `p` в степени;
- делать много лишних умножений.

Метод Горнера переписывает вычисление в более удобную вложенную форму.

Например:

```text
s[0] + p(s[1] + p(s[2] + ... ))
```

или в эквивалентной ориентации:

```text
((s[n-1] * p + s[n-2]) * p + ...)
```

Главная идея одна:

> не считать степени отдельно, а накапливать значение последовательно.

## 9. Почему метод Горнера хорош

### 9.1. Меньше операций

Не нужно отдельно вычислять `p^i`.

### 9.2. Линейное вычисление

Хеш можно посчитать за один проход по строке.

### 9.3. Простой код

Алгоритм компактный и хорошо реализуется.

## 10. Реализация на C++

```cpp
long long PolyHash(const std::string& s, long long p, long long mod) {
  long long hash = 0;
  long long power = 1;
  for (char ch : s) {
    hash = (hash + (static_cast<long long>(ch) * power) % mod) % mod;
    power = (power * p) % mod;
  }
  return hash;
}
```

Или более “горнеровский” стиль:

```cpp
long long PolyHashHorner(const std::string& s, long long p, long long mod) {
  long long hash = 0;
  for (char ch : s) {
    hash = (hash * p + static_cast<long long>(ch)) % mod;
  }
  return hash;
}
```

Оба варианта полезны, просто по-разному организуют счёт.

## 11. Префиксные хеши: почему полиномиальный хеш особенно силён

Настоящая мощь темы раскрывается, когда нужно быстро сравнивать подстроки.

Если заранее посчитать:

- хеши всех префиксов;
- степени `p`,

то можно получать хеш подстроки очень быстро.

Это превращает строку в объект, над которым можно делать много запросов почти за
константу.

## 12. Префиксные хеши

Пусть:

```text
pref[i] = хеш префикса длины i
```

Тогда хеш подстроки можно выразить через разность двух префиксов с поправкой на
степень `p`.

Именно эта алгебра делает возможными быстрые строковые алгоритмы.

## 13. Где это особенно применяется

### 13.1. Рабин — Карп

Сравнение подстрок начинается с сравнения их хешей.
Если хеши разные — строки точно разные.
Если одинаковые — возможно совпадение, иногда требуется дополнительная
проверка.

### 13.2. Сравнение подстрок

Можно быстро отвечать на вопросы:

- равны ли две подстроки;
- где встречается шаблон;
- как сравниваются фрагменты текста.

### 13.3. Строковые структуры

Полиномиальный хеш часто идёт рядом с:

- суффиксными массивами;
- бинарным поиском по ответу;
- задачами на повторы и палиндромы.

## 14. Почему коллизии всё равно возможны

Полиномиальный хеш — не идеален.
Разные строки могут дать одинаковое значение.

Это называется коллизией.

Причина та же, что и в обычном хешировании:

- множество строк огромно;
- диапазон остатков по модулю конечен.

## 15. Как уменьшают риск коллизий

Обычно используют:

- большой модуль;
- хорошее основание `p`;
- два разных модуля;
- двойной хеш.

Если два независимых хеша совпали одновременно, вероятность случайной коллизии
становится значительно меньше.

## 16. Почему полиномиальный хеш особенно хорош именно на строках

Потому что он одновременно:

- чувствителен к порядку символов;
- хорошо обновляется;
- поддерживает префиксную алгебру;
- допускает быстрые сравнения подстрок.

Это не просто “одна из хеш-функций”, а почти базовый язык для алгоритмов на
строках.

## 17. Ограничения и осторожность

Нельзя слепо полагаться на хеш как на абсолютное доказательство равенства
строк, если требуется строгая корректность.

Часто:

- хеш используется как быстрый фильтр;
- а при совпадении выполняется дополнительная проверка;
- или применяют двойной хеш для практической надёжности.

## 18. Типичные ошибки

- считать, что одинаковый хеш всегда означает одинаковые строки;
- выбирать слишком маленький модуль;
- не различать направление вычисления в формуле;
- забывать про степени `p` при работе с подстроками;
- путать полиномиальный хеш для строк с контейнерным хешированием ключей.

## 19. Что важно запомнить

Полиномиальный хеш — это один из важнейших инструментов строковых алгоритмов.

Главные идеи:

1. строка трактуется как полином;
2. порядок символов учитывается автоматически;
3. метод Горнера позволяет считать хеш быстро;
4. префиксные хеши открывают доступ к быстрым операциям над подстроками;
5. коллизии возможны, поэтому важны параметры и инженерная осторожность.

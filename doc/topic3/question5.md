# Поиск ключа, вставка, удаление

## 1. Почему именно эти операции считаются базовыми

Когда мы говорим, что дерево поиска — это контейнер, мы почти всегда имеем в
виду три базовые операции:

- `find(k)` — найти ключ;
- `insert(k)` — добавить ключ;
- `erase(k)` — удалить ключ.

Именно они определяют, действительно ли структура годится для динамического
множества.

---

## 2. Поиск: самый естественный алгоритм в BST

Поиск устроен почти очевидно:

1. сравниваем искомый ключ `k` с ключом текущей вершины `x`;
2. если `k = x`, поиск завершён;
3. если `k < x`, идём в левое поддерево;
4. если `k > x`, идём в правое поддерево.

### Почему это корректно

Потому что инвариант `BST` гарантирует:

- всё левое поддерево состоит только из меньших ключей;
- всё правое — только из больших.

Значит после одного сравнения мы можем безопасно отбросить половину
“структурного пространства”.

---

## 3. Поиск — это движение по одному пути

Очень важно понимать асимптотику не формально, а геометрически:

- поиск никогда не обходит оба поддерева одновременно;
- он всегда идёт только по одному пути от корня вниз.

Поэтому:

```text
find = O(h)
```

где `h` — высота дерева.

---

## 4. Вставка как продолжение поиска

Вставка не придумывает новую логику — она использует ту же идею, что и поиск.

Мы:

1. идём вниз, сравнивая новый ключ с текущими;
2. находим место, где должен был бы лежать этот ключ;
3. если там пусто, создаём новую вершину.

То есть вставка — это “поиск позиции для нового элемента”.

---

## 5. Почему вставка сохраняет инвариант

Когда мы вставляем новый ключ `k`, путь до места вставки уже полностью
определён сравнениями:

- если мы ушли влево, значит `k` меньше всех вершин, которые заставили нас туда
  уйти;
- если ушли вправо, значит больше соответствующих вершин.

Следовательно, в найденной пустой позиции новый ключ автоматически попадает в
правильное место относительно всех предков.

Именно поэтому обычная BST-вставка так красива: инвариант сохраняется почти
сам собой.

---

## 6. Что делать с дубликатами

Тут важно заранее выбрать политику.

### Вариант 1. Запрещаем дубликаты

Если нашли существующий ключ, ничего не вставляем.

### Вариант 2. Кладём равные всегда в одну сторону

Например, всегда вправо.

### Вариант 3. Храним счётчик в вершине

Это часто самый аккуратный способ, если структура рассматривается как
мультимножество.

Главное правило:

политика должна быть единообразной, иначе дерево перестанет быть корректным.

---

## 7. Удаление: почему это самая тонкая операция

Поиск и вставка в `BST` почти полностью следуют из инварианта порядка.

Удаление сложнее, потому что нужно сделать сразу две вещи:

- убрать вершину;
- не сломать структуру поиска.

То есть задача уже не только “найти место”, а ещё и “правильно перестроить
связи”.

---

## 8. Три стандартных случая удаления

### Случай 1. Вершина — лист

У неё нет детей.

Тогда удаление тривиально:

- просто отцепляем её от родителя.

### Случай 2. У вершины один ребёнок

Тогда ребёнок поднимается на её место.

Это безопасно, потому что всё поддерево этого ребёнка уже лежит в правильном
интервале ключей.

### Случай 3. У вершины два ребёнка

Это главный содержательный случай.

Просто удалить вершину нельзя, потому что она разделяла:

- все ключи слева;
- все ключи справа.

Нужно заменить её вершиной, которая:

- максимально близка к ней по порядку;
- и не разрушит BST-инвариант.

---

## 9. Почему берут предшественника или преемника

Обычно выбирают одну из двух вершин:

- **предшественник** — максимум из левого поддерева;
- **преемник** — минимум из правого поддерева.

Почему именно они:

- предшественник — это самый большой ключ, который всё ещё меньше удаляемого;
- преемник — самый маленький ключ, который всё ещё больше удаляемого.

То есть это ровно те вершины, которые “лучше всего подходят” на замену по
порядку.

---

## 10. Почему после замены остаётся простой случай

Очень красивый факт:

если взять преемника или предшественника, то вершина-замена почти всегда
находится в более простом положении.

Например, минимум правого поддерева:

- не может иметь левого ребёнка,
  потому что иначе слева был бы ещё меньший элемент.

Значит после копирования/перемещения ключа мы сводим задачу к удалению вершины
с не более чем одним ребёнком.

Именно так удаление с двумя детьми сводится к уже понятному более простому
случаю.

---

## 11. Визуальный пример удаления

Пусть нужно удалить `8`:

```text
        8
      /   \
     3    10
    / \     \
   1   6     14
      / \    /
     4   7  13
```

Минимум правого поддерева — это `10`.

Тогда можно:

1. заменить `8` на `10`;
2. удалить старую вершину `10` из правого поддерева.

После этого порядок по ключам сохраняется.

---

## 12. Откуда берётся сложность операций

Во всех трёх операциях основная работа идёт по одному пути вниз:

- поиск идёт к нужному ключу;
- вставка идёт к месту вставки;
- удаление сначала ищет вершину, а потом ещё делает локальную перестройку.

Поэтому:

```text
find   = O(h)
insert = O(h)
erase  = O(h)
```

где `h` — высота дерева.

Если дерево хорошей формы:

```text
O(log n)
```

если выродилось:

```text
O(n)
```

## 13. Реализация на C++

```cpp
struct Node {
  int key;
  Node* left = nullptr;
  Node* right = nullptr;
};

Node* Find(Node* root, int key) {
  if (root == nullptr || root->key == key) {
    return root;
  }
  if (key < root->key) {
    return Find(root->left, key);
  }
  return Find(root->right, key);
}

Node* FindMin(Node* root) {
  while (root != nullptr && root->left != nullptr) {
    root = root->left;
  }
  return root;
}

Node* FindMax(Node* root) {
  while (root != nullptr && root->right != nullptr) {
    root = root->right;
  }
  return root;
}

Node* Insert(Node* root, int key) {
  if (root == nullptr) {
    return new Node{key};
  }
  if (key < root->key) {
    root->left = Insert(root->left, key);
  } else if (key > root->key) {
    root->right = Insert(root->right, key);
  }
  return root;
}

Node* Erase(Node* root, int key) {
  if (root == nullptr) {
    return nullptr;
  }

  if (key < root->key) {
    root->left = Erase(root->left, key);
    return root;
  }
  if (key > root->key) {
    root->right = Erase(root->right, key);
    return root;
  }

  if (root->left == nullptr) {
    Node* right = root->right;
    delete root;
    return right;
  }
  if (root->right == nullptr) {
    Node* left = root->left;
    delete root;
    return left;
  }

  Node* successor = FindMin(root->right);
  root->key = successor->key;
  root->right = Erase(root->right, successor->key);
  return root;
}
```

## 14. Рекурсивное мышление о BST-операциях

Очень удобно смотреть на все три операции рекурсивно.

### Поиск

“Если ключ меньше, ищем в левом поддереве; если больше — в правом.”

### Вставка

“Если ключ должен лежать слева, вставляем в левое поддерево; если справа —
в правое.”

### Удаление

“Если удаляемый ключ меньше — удаляем в левом поддереве; если больше — в
правом; если нашли текущую вершину — разбираем её локальный случай.”

Это очень важно, потому что почти все самобалансирующиеся деревья потом строят
свои вставки и удаления именно поверх такой базовой рекурсивной схемы.

---

## 15. Почему удаление ощущается сложнее психологически

Не потому, что оно принципиально “другой природы”, а потому, что там есть два
слоя логики:

1. найти вершину по правилу `BST`;
2. правильно перестроить связи после её удаления.

Если удерживать эту двухслойную картину в голове, операция становится намного
понятнее.

---

## 16. Типичные ошибки

### Ошибка 1. Сравнивать вершину только с родителем

Нужно сохранять корректность относительно всего пути предков.

### Ошибка 2. Удалить вершину с двумя детьми “простым удалением”

Нельзя, иначе ломается структура поиска.

### Ошибка 3. Не продумать политику для равных ключей

Дубликаты без явной договорённости быстро ломают инвариант.

### Ошибка 4. Думать, что операции всегда `O(log n)`

Нет. Только если высота дерева действительно логарифмическая.

---

## 17. Что важно запомнить

Операции `BST` красивы потому, что почти полностью следуют из одного
инварианта порядка.

Нужно особенно хорошо понимать:

1. поиск как спуск по сравнению;
2. вставку как поиск пустого места;
3. удаление через три случая;
4. замену на предшественника или преемника;
5. зависимость всех оценок от высоты дерева.

Если эти идеи ясны, то дальше самобалансирующиеся деревья воспринимаются как:

> обычные операции `BST` плюс механизм удержания высоты под контролем.
